# gen-github-workflow

Code-generation tool that scans source files for embedded workflow fragments and assembles them into complete GitHub Actions workflow YAML files.

## Motivation

Workflow definitions grow large and repetitive. By co-locating small workflow fragments next to the code they relate to, teams can keep CI concerns close to the source while a single generator merges everything into valid `.github/workflows/*.yml` files.

## How it works

### 1. Scan

Recursively walk a **base folder** (default: repo root) and find every file that contains the marker comment:

```
//<gen-github-workflow>
```

Everything between a `<gen-github-workflow>` opening tag and its corresponding `</gen-github-workflow>` closing tag is treated as a **YAML fragment**. A single file may contain multiple fragments.

```ts
// some-service.ts

//<gen-github-workflow>
// name: ci-some-service
// on:
//   push:
//     branches: [main]
// jobs:
//   build:
//     steps:
//       - uses: actions/checkout@v5
//</gen-github-workflow>
```

Comment prefixes (`//`, `#`, etc.) at the start of each line inside the block are stripped before parsing (see **Comment stripping** below).

### 2. Parse

Each extracted block is parsed as YAML. The parser must:

- Strip leading comment characters (`//` or `#`) and an optional single space from every line.
- **Expand template variables** (see **Template variables** below).
- Validate that the result is valid YAML.
- Require a top-level `name` key to identify the workflow.

### Template variables

Before YAML parsing, the following template variables are expanded in each fragment:

| Variable | Expands to | Example |
|----------|------------|--------|
| `$$file` | Relative path from repo root to the source file | `apps/api/src/server.ts` |
| `$$directory` | Relative path from repo root to the source file's directory | `apps/api/src` |

These are useful for `paths:` triggers so fragments can reference their own location without hardcoding:

```ts
//<gen-github-workflow>
// name: ci-api
// on:
//   push:
//     paths:
//       - "$$directory/**"
//</gen-github-workflow>
```

If the file is at `apps/api/src/server.ts`, `$$directory` expands to `apps/api/src`.

Template variables are expanded **before** YAML parsing, so they work in any value position including inside quoted strings.

### 3. Accumulate / Merge

Fragments that share the same `name` are **deep-merged** into a single workflow document. Merge rules:

| Key | Strategy |
|-----|----------|
| `on` (triggers) | union of all trigger definitions |
| `env` | shallow merge, later wins |
| `permissions` | shallow merge, later wins |
| `concurrency` | last writer wins |
| `jobs` | merge by job id; within a job, `steps` arrays are concatenated |

If two fragments for the same workflow have conflicting scalar values for the same key (outside the rules above), emit a **warning** and let the last-seen fragment win.

### 4. Apply defaults

Before writing, inject default values into jobs:

- **`runs-on`**: Jobs that do not specify `runs-on` and do not use `uses:` (reusable workflow calls) receive `runs-on: ubuntu-latest`.

This keeps fragments concise — most jobs should omit `runs-on` and inherit the default.

### 5. Output

Write each merged workflow to the **output folder** (default: `.github/workflows/`) as `<name>.yml`.

If a generated file would overwrite a file that was **not** previously generated by this tool, abort with an error. Generated files carry a header comment:

```yaml
# AUTO-GENERATED by gen-github-workflow -- DO NOT EDIT
# Source fragments found via //<gen-github-workflow> markers
```

## CLI

Entrypoint: `src/index.ts` (bun/ts). It is run directly with Bun.

```
Usage: gen-github-workflow <command> [options]

Commands:
  generate           Scan, merge, and write workflow files (default command)
  check              Dry-run: exit 1 if generated output differs from existing files
  e2e-test           Run e2e tests against fixtures

Options:
  --base <path>      Base folder to scan (default: repo root, detected via git)
  --output <path>    Output folder for generated workflows (default: <base>/.github/workflows)
  --verbose          Print discovered fragments and merge diagnostics
  --help             Show help
```

### `check` command

Useful in CI to verify that committed workflows are up-to-date with their source fragments. Exits 0 if everything matches, 1 if there are differences (with a diff printed to stderr).

### `generate` command

Runs the full pipeline: scan → parse → merge → output. Writes generated workflow files to the output folder.

## E2E testing

`index.ts` exposes an **e2e-test** command (e.g. `bun run index.ts e2e-test`) that:

1. Runs the generator against `e2e-test/` as the base folder.
2. Compares output to `e2e-test-expected/`.
3. Exits 0 on match, 1 on diff (printing the diff).

This lets us iterate on the generator with fast, deterministic feedback.

## CI macros

`src/index.ts` contains `//<gen-github-workflow>` blocks that define two CI workflows:

#### `check-generated-workflows`

Runs on **every push and pull request**. Executes `gen-github-workflow check` to verify that committed `.github/workflows/` files are up-to-date with their source fragments. Fails if any generated file is stale.

Uses `$$directory` to scope the paths trigger to the tool's own directory.

#### `e2e-test-gen-github-workflow`

Runs on **push and pull request when `$$directory/**` changes**. Executes `gen-github-workflow e2e-test` to validate the tool against its fixture suite. Ensures the generator itself is working correctly whenever its source is modified.

## Edge cases & decisions

- **Nested markers**: not supported. If `<gen-github-workflow>` appears inside an already-open block, emit an error.
- **Empty blocks**: skip silently.
- **Encoding**: UTF-8 only.
- **Binary files**: skip files that fail UTF-8 decode.
- **Symlinks**: follow symlinks (default `fs` behavior).
- **Ignore patterns**: respect `.gitignore` via git ls-files or a glob library. Always skip `node_modules/`, `.git/`, and the output folder itself.
- **Template variables in non-repo context**: if the tool cannot detect a git repo root (e.g. when scanning a standalone directory), `$$file` and `$$directory` resolve relative to the `--base` folder instead.
